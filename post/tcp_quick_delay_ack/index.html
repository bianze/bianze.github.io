引言

之前有相关的patch将TCP_DELACK_MIN和TCP_DELACK_MAX通过sysctl方式export到proc文件系统，通过sysctl命令动态地修改这个值，而且在我们的场景中也偶尔会看到delay ack回复慢的情况造成业务抖动，那么我们就来分析看看这个patch是否能解决我们的问题。patch链接如下：

https://patchwork.ozlabs.org/project/netdev/patch/1319756841-2051-1-git-send-email-dbaluta@ixiacom.com/

要回答这个问题，那么我们就要先从代码层面分析什么场景下tcp的delay ack会超时？只有了解了触发这个问题的场景，我们再来看在我们的场景中是否有必要对这个超时时间进行调控，进而再分析如何调控这个超时时间才能达到我们想要的效果。

TCP ACK机制简介

在具体分析之前我们有必要了解什么是quick ack, 什么是delay ack。

当TCP连接刚建立的时候，拥塞控制处于慢启动状态，需要根据接收端回复的ACK来增加拥塞窗口，因此这个时候ACK的回复快慢对慢启动的时间就会造成很大的影响，在这个阶段就有必要对每个收到的报文进行快速的ACK确认，当然quick ack状态不仅在这一个时候会触发，在其它的一些条件也会进入快速进行ack确认的情况，具体的下面可以根据源码进行解读。

如果持续的进行每收到一个包就回复一个ACK的话会增加ACK的数量，这样占用一定的CPU和网络资源，因此可以适当延缓ACK的回复，当接收端也有数据发送的时候，就可以考虑将数据和ACK一起发送到发送端，这样的话可以节省资源。这种发送端发送一些数据到接收端，接收端同时也有数据发送的场景叫乒乓场景，进入这种状态后会将icsk->icsk_ack.pingpong置为1。当然延迟ACK的回复也不是说只有接收端也有数据时才会发送，还是有一些其它条件可以触发的。当没有任何条件出发delay ack回复时，delay ack就会等到设定的超时时间后触发一次ack回复。我们今天要分析的就是在什么情况下delay ack会进入到超时发送的流程，并且分析这个超时时间是否可以进行合理调控后让它能够更加快速的回复。

一句话总结就是quick ack机制为了更好的进行拥塞控制，delay ack机制为了更好地利用网络资源。

/* Send ACKs quickly, if "quick" count is not exhausted
 * and the session is not interactive.
 */

static bool tcp_in_quickack_mode(struct sock *sk)
{
        const struct inet_connection_sock *icsk = inet_csk(sk);
        const struct dst_entry *dst = __sk_dst_get(sk);

        return (dst && dst_metric(dst, RTAX_QUICKACK)) ||
                (icsk->icsk_ack.quick && !icsk->icsk_ack.pingpong); //quick计数还没有耗尽并且不处于乒乓模式，也就是处于乒乓模式的话就不会进行快速确认；quick计数在第一个数据包收到时会设定一个初始值，后续交互到一定时候会设置成0
}

void tcp_send_delayed_ack(struct sock *sk)
{
        struct inet_connection_sock *icsk = inet_csk(sk);
        int ato = icsk->icsk_ack.ato;
        unsigned long timeout;

        if (ato > TCP_DELACK_MIN) {
                const struct tcp_sock *tp = tcp_sk(sk);
                int max_ato = HZ / 2;
    
                if (icsk->icsk_ack.pingpong ||
                    (icsk->icsk_ack.pending & ICSK_ACK_PUSHED))
                        max_ato = TCP_DELACK_MAX;
    
                /* Slow path, intersegment interval is "high". */
    
                /* If some rtt estimate is known, use it to bound delayed ack.
                 * Do not use inet_csk(sk)->icsk_rto here, use results of rtt measurements
                 * directly.
                 */
                if (tp->srtt_us) {
                        int rtt = max_t(int, usecs_to_jiffies(tp->srtt_us >> 3),
                                        TCP_DELACK_MIN);
    
                        if (rtt < max_ato)
                                max_ato = rtt;
                }
    
                ato = min(ato, max_ato);
        }
    
        /* Stay within the limit we were given */
        timeout = jiffies + ato;//设定delay ack定时器的超时时间，这个时间是根据ato的值来的
  			//总结下来，ato = min(icsk->icsk_ack.ato, min(TCP_DELACK_MAX, max(rtt, TCP_DELACK_MIN)))
    
            /* Use new timeout only if there wasn't a older one earlier. */
            if (icsk->icsk_ack.pending & ICSK_ACK_TIMER) {
                    /* If delack timer was blocked or is about to expire,
                     * send ACK now.
                     */
                    if (icsk->icsk_ack.blocked ||
                        time_before_eq(icsk->icsk_ack.timeout, jiffies + (ato >> 2))) {
                            tcp_send_ack(sk);
                            return;
                    }
    
                    if (!time_before(timeout, icsk->icsk_ack.timeout))
                            timeout = icsk->icsk_ack.timeout;
            }
            icsk->icsk_ack.pending |= ICSK_ACK_SCHED | ICSK_ACK_TIMER;
            icsk->icsk_ack.timeout = timeout;
            sk_reset_timer(sk, &icsk->icsk_delack_timer, timeout); //设置定时器超时时间
}

static void tcp_event_data_recv(struct sock *sk, struct sk_buff *skb)
{
    struct tcp_sock *tp = tcp_sk(sk);
    struct inet_connection_sock *icsk = inet_csk(sk);
    u32 now;

    inet_csk_schedule_ack(sk);

    tcp_measure_rcv_mss(sk, skb);

    tcp_rcv_rtt_measure(tp);

    now = tcp_jiffies32;

    if (!icsk->icsk_ack.ato) {
            /* The _first_ data packet received, initialize
             * delayed ACK engine.
             */
            tcp_incr_quickack(sk, TCP_MAX_QUICKACKS);//收到第一个数据包，设置quick的计数
            icsk->icsk_ack.ato = TCP_ATO_MIN; //设置ato的初始值为TCP_ATO_MIN
    } else {
            int m = now - icsk->icsk_ack.lrcvtime;//收包间隔

            if (m <= TCP_ATO_MIN / 2) {
                    /* The fastest case is the first. */
                    icsk->icsk_ack.ato = (icsk->icsk_ack.ato >> 1) + TCP_ATO_MIN / 2;//1.当收包间隔小于TCP_ATO_MIN/2时,ato初始值为TCP_ATO_MIN
            } else if (m < icsk->icsk_ack.ato) {//2.收包间隔大于TCP_ATO_MIN/2小于ato时，此时m > TCP_ATO_MIN/2，结合1，2我们可以确认ato的值是始终大于TCP_ATO_MIN无疑了
                    icsk->icsk_ack.ato = (icsk->icsk_ack.ato >> 1) + m;
                    if (icsk->icsk_ack.ato > icsk->icsk_rto)
                            icsk->icsk_ack.ato = icsk->icsk_rto;
            } else if (m > icsk->icsk_rto) {//收包间隔大于超时时间，保持ato的值不变，进入quick ack模式
                    /* Too long gap. Apparently sender failed to
                     * restart window, so that we send ACKs quickly.
                     */
                     tcp_incr_quickack(sk, TCP_MAX_QUICKACKS);
                        sk_mem_reclaim(sk);
                }
        }
        icsk->icsk_ack.lrcvtime = now;

        tcp_ecn_check_ce(sk, skb);

        if (skb->len >= 128)
                tcp_grow_window(sk, skb);
}

结合min(icsk->icsk_ack.ato, min(TCP_DELACK_MAX, max(rtt, TCP_DELACK_MIN)))公式和这段代码的分析我们可以确认delay ack的超时间隔，肯定是介于min(TCP_ATO_MIN, TCP_DELACK_MIN)和TCP_DELACK_MAX之间的，而现在系统的宏TCP_ATO_MIN=TCP_DELACK_MIN=40ms, TCP_DELACK_MAX=200ms，也就是说如果我们需要调控的话需要同时调控TCP_ATO_MIN, TCP_DELACK_MIN和TCP_DELACK_MAX，而上述链接中对sysctl_tcp_delack_segs的调控，只会减缓de lay ack回复的速度，对于我们希望加快delay ack回复的场景没有帮助。

但是我们还是没能解释清楚，什么情况下就会触发delay ack超时，触发了delay ack超时后对业务到底又有什么影响呢？

这个时候我们就需要跟发送端结合起来进行分析，试想在__tcp_ack_snd_check中，如果接收到的包没有超过一个mss，tcp又处于乒乓模式，这样就进入了delay ack模式，如果这个时候会导致发送端无法继续发送数据，需要等到delay ack超时后回复ack到发送端，然后才有数据发送过来，是不是就出问题了？那这种场景存不存在呢，答案是肯定的，真就存在这种场景。

linux tcp协议栈默认就开启了nagle算法，也就是说tcp协议栈默认会做的事情就是将小报文聚合成一个mss的报文，然后再发送给接收端，当然这个聚合也是要满足一定的条件的，而如果要禁用掉这个算法的话，需要针对套接字设定套接字选项，通过设置TCP_NODELAY把nagle算法禁掉，如果开启了TCP_NODELAY当然是不会触发我们说的上述问题的。关键在于很多情况业务应该不会特地去设这个选项，但又发送小报文，看下tcp发端这部分代码的处理.

/* Return true if the Nagle test allows this packet to be
 * sent now.
 */
   static inline bool tcp_nagle_test(const struct tcp_sock *tp, const struct sk_buff *skb,
                                 unsigned int cur_mss, int nonagle)
   {
       /* Nagle rule does not apply to frames, which sit in the middle of the
        * write_queue (they have no chances to get new data).
        *
        * This is implemented in the callers, where they modify the 'nonagle'
        * argument based upon the location of SKB in the send queue.
        */
       if (nonagle & TCP_NAGLE_PUSH)//1.这个时候nonagle基本都是0，这个条件不成立
               return true;

       /* Don't use the nagle rule for urgent data (or for the final FIN). */
       if (tcp_urg_mode(tp) || (TCP_SKB_CB(skb)->tcp_flags & TCPHDR_FIN))//2.需要紧急数据发送或者发送FIN报文，不成立
               return true;
       
       if (!tcp_nagle_check(skb->len < cur_mss, tp, nonagle)) //3.判定nagle算法是否生效，生效的话就会走到最后tcp_nagle_test返回false，当前这个包不能立即发送
               return true;
       
       return false;

   }

/* Return false, if packet can be sent now without violation Nagle's rules:

 * 1. It is full sized. (provided by caller in %partial bool)
 * 2. Or it contains FIN. (already checked by caller)
 * 3. Or TCP_CORK is not set, and TCP_NODELAY is set.
 * 4. Or TCP_CORK is not set, and all sent packets are ACKed.
 * With Minshall's modification: all sent small packets are ACKed.
 */
   static bool tcp_nagle_check(bool partial, const struct tcp_sock *tp,
                         int nonagle)
   {
     return partial && //可以看到发送的数据小于mss时，partial为真
             ((nonagle & TCP_NAGLE_CORK) || //如果显式设置了TCP_CORK的话，肯定是要塞住的，这个flag是通过setsockopt设置的，一般时候应用也不会去设
              (!nonagle && tp->packets_out && tcp_minshall_check(tp))); //nonagle为0，并且还有数据包发出去了未被ack，并且上一个发送的小包没有确认，这三个条件在小报文发送的场景是很有可能一起满足的！！！
   }

结论与解法

通过上面的分析我们应该能够清晰的看到，如果业务层面是在进行小包发送，且收端也偶尔有数据需要回复给发端，这时候进入乒乓模式，而发端上一个发送的小报文出去后，收端进入delay ack确认流程，这个时候业务再发送小报文到协议栈，协议栈则不能立即将报文发送出去了，需要等待delay ack超时后回复延迟ack才能再进一步发送数据。

这么看来是有业务场景很可能会触发这个问题的，而且在数据中心网络里面，rtt是很小的，如果要等到delay ack超时的话，当前至少是40ms的时间，那么这个时延对于业务来说是不可接受的，因此我们还是很有必要对这个超时时间进行调控。

调控的方法就是需要将TCP_ATO_MIN, TCP_DELACK_MIN, TCP_DELACK_MAX通过sysctl export出来，调控时尽量让sysctl_tcp_ato_min = sysctl_tcp_delack_min，我觉得可以尝试定位4ms，sysctl_tcp_delack_max可以设定成40ms, 在数据中心网络里面rtt很小，因此大部分超时时间应该可以控制在4ms多一点，这个需要等待实践证明。
